## 🗄️ DATABASE SCHEMA (Supabase SQL)

```sql
-- Profiles table
create table profiles (
  id uuid references auth.users primary key,
  full_name text,
  username text unique,
  title text,
  bio text,
  avatar_url text,
  location text,
  current_company text,
  current_role text,
  work_status text default 'employed', -- 'employed' | 'freelance' | 'open'
  updated_at timestamptz default now()
);

-- Connected platforms table (one row per connected account)
create table connected_platforms (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references profiles(id),
  platform text not null, -- 'github' | 'linkedin' | 'twitter' | 'slack'
  platform_user_id text,
  platform_username text,
  access_token text, -- encrypted via Supabase Vault
  refresh_token text,
  token_expires_at timestamptz,
  scopes text[], -- what permissions were granted
  is_active boolean default true,
  connected_at timestamptz default now(),
  last_synced_at timestamptz,
  unique(user_id, platform)
);

-- Activity feed table (cached data from all platforms)
create table activity_feed (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references profiles(id),
  platform text not null,
  activity_type text, -- 'commit' | 'repo' | 'post' | 'tweet' | 'message'
  title text,
  description text,
  url text,
  media_url text,
  metadata jsonb, -- platform-specific extra data
  activity_at timestamptz,
  synced_at timestamptz default now()
);

-- Projects table
create table projects (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references profiles(id),
  name text not null,
  description text,
  tech_stack text[],
  status text default 'in_progress', -- 'live' | 'in_progress' | 'open_source' | 'archived'
  github_repo text, -- e.g. "username/repo-name" for GitHub API linking
  demo_url text,
  cover_image text,
  is_featured boolean default false,
  sort_order int default 0,
  created_at timestamptz default now()
);

-- Enable RLS
alter table profiles enable row level security;
alter table connected_platforms enable row level security;
alter table activity_feed enable row level security;
alter table projects enable row level security;

-- RLS Policies (owner only)
create policy "Users can manage own profile" on profiles for all using (auth.uid() = id);
create policy "Users can manage own platforms" on connected_platforms for all using (auth.uid() = user_id);
create policy "Users can manage own activity" on activity_feed for all using (auth.uid() = user_id);
create policy "Users can manage own projects" on projects for all using (auth.uid() = user_id);
-- Public read for portfolio display
create policy "Public can view projects" on projects for select using (true);
create policy "Public can view profiles" on profiles for select using (true);
```

## 🔐 OAUTH INTEGRATION PER PLATFORM

### How it works
The user visits /settings/connections and connects each platform one by one.
Supabase handles the OAuth dance and stores provider_token in the session.
After OAuth, a Supabase Edge Function is called to:
1. Save the token to connected_platforms table (encrypted via Vault)
2. Do an initial sync of activity data → activity_feed table

### GitHub Integration
- Supabase built-in GitHub OAuth provider → enable in Supabase dashboard
- Scopes needed: `read:user`, `repo` (for private), `public_repo`, `read:org`
- After connect → call GitHub API:
  - GET /user → profile info
  - GET /users/{username}/repos → repos list
  - GET /users/{username}/events → recent activity (push, pr, issues)
  - GET https://github-contributions-api.jogruber.de/v4/{username} → contribution data

### X (Twitter) Integration
- Use Twitter OAuth 2.0 with PKCE
- Required scopes: `tweet.read`, `users.read`, `offline.access`
- Supabase has built-in Twitter provider → enable in dashboard
- After connect → call X API v2:
  - GET /2/users/me → user info
  - GET /2/users/:id/tweets?max_results=10 → recent tweets (Free tier: 500/month)
  - Note: Display a "Last synced: X ago" and sync manually (not auto) to save API quota

### LinkedIn Integration  
- LinkedIn OAuth 2.0 via Supabase custom OIDC provider
- Required scopes: `openid`, `profile`, `email`, `w_member_social`
- After connect → call LinkedIn API v2:
  - GET /v2/me → profile
  - GET /v2/ugcPosts?q=authors → recent posts (requires Marketing Developer Platform approval)
  - FALLBACK: If API not approved yet, show "Manual post" option where user can paste LinkedIn post URL and we scrape/store it manually

### Slack Integration
- Slack OAuth 2.0 (user token, not bot token)
- Scopes: `users:read`, `channels:history`, `im:history`
- Not in Supabase built-in → use custom OAuth flow via Supabase Edge Function
- After connect: show workspace name, recent public messages/activity

## 📋 IMPORTANT IMPLEMENTATION NOTES

1. **Token Security:** NEVER store OAuth tokens in Pinia/localStorage. 
   Store only in `connected_platforms` table server-side. 
   All API calls that need the token go through Supabase Edge Functions.

2. **LinkedIn API Limitation:** LinkedIn's ugcPosts endpoint for reading your own posts 
   requires "Marketing Developer Platform" status which needs LinkedIn approval.
   Build a graceful fallback: manual post entry (title, URL, date, platform=linkedin).

3. **X/Twitter Free Tier:** 500 tweet reads/month. 
   Cache all fetched tweets in activity_feed. 
   Add a cooldown: only allow manual sync once every 6 hours.

4. **GitHub is the most open:** Use it as the primary activity source. 
   The contributions API (https://github-contributions-api.jogruber.de/v4/{username}) 
   is public and doesn't require auth — use it for the heatmap widget.

5. **Supabase Realtime:** Subscribe to `activity_feed` changes in the dashboard 
   so when an edge function completes a sync, the UI updates live without refresh.

6. **Public vs Private data:** 
   - Public portfolio shows: project names, skills, current employer, last 5 activity items (no token data)
   - Private dashboard shows: full activity, connection status, token health, sync controls

7. **Progressive OAuth:** Don't force connecting all platforms. 
   Each platform is optional. Show a "Not connected" state gracefully on the portfolio.
